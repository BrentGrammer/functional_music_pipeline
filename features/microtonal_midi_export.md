# Microtonal MIDI Export Design

## Summary
The goal is to extend the existing MIDI export functionality to support microtonal tuning. By default, standard MIDI 1.0 does not support floating-point note numbers. To preserve the exact frequencies generated by algorithmic transforms (which may land between standard 12-TET semitones), we will use MIDI Pitch Bend.

2. **Per-Voice Channel Assignment**
   - Pitch Bend is a channel-wide message. To allow independent microtonal melodies, each monophonic `Voice` in the `Score` will be assigned its own MIDI channel.
   - **Requirement:** This approach assumes each `Voice` remains monophonic (one note at a time). If polyphonic voices are needed later, we would need to explore MPE or more complex channel allocation.
   - We will utilize MIDI channels 0-15.
   - **Important:** Channel 9 (10 in 1-indexed DAWs) is skipped. **Why:** Many DAWs and VSTs hardcode Channel 9 for percussion. Sending melodic data or Pitch Bend Range (RPN) messages to Channel 9 often results in unexpected drum sounds or ignored tuning data.
   - **Exceeding 15 Voices ("Zero and Fallback" Strategy):**
     - Because MIDI only has 15 usable channels (skipping 9), we cannot support independent pitch bends for 16 or more voices.
     - If a `Score` contains more than 15 voices, voices 16 and beyond will be assigned to a fallback channel (e.g., wrapped to Channel 0) and their `offset_cents` will be forced to `0`.
     - This guarantees no "pitch glitching" between shared channels, at the cost of losing microtonal accuracy for the excess voices.
     - We will print a console warning informing the user that voices > 15 have been rounded to standard tuning.
   - **Implementation Steps:**
     - 2.1 Refactor `save_score_to_midi` loop to track channel assignments, skipping Channel 9.
     - 2.2 Add `channel` argument to `voice_track.append(Message(...))` calls.
     - 2.3 Implement the "Zero and Fallback" logic: If `voice_index >= 15`, wrap the channel and force the calculated `offset_cents` to `0.0`.
     - 2.4 Add logic to print a single warning to the console if `len(score.voices) > 15`.
     - 2.5 Add tests to verify that channel 9 is skipped, `offset_cents` are zeroed for voices > 15, and the warning is triggered.

3. **Explicit Pitch Bend Sensitivity**
   - To ensure DAWs like Cubase and their VST instruments interpret the pitch bend values correctly, we will explicitly set the Pitch Bend Range for each channel at the beginning of its track.
   - We will use standard Registered Parameter Number (RPN) messages to set the bend range to exactly **±1 semitone**.
   - **Note:** This tuning only sounds correct if the target VST or synth respects the RPN 0 pitch bend range message.
   - Sequence:
     - CC 101 (RPN MSB): 0
     - CC 100 (RPN LSB): 0
     - CC 6 (Data Entry MSB): 1 (for 1 semitone)
     - CC 38 (Data Entry LSB): 0 (for 0 cents)
   - **Implementation Steps:**
     - 3.1 Extract RPN message generation into a helper function `_initialize_pitch_bend_range(channel: int)`.
     - 3.2 Call this helper at the beginning of each track creation in `save_score_to_midi` to initialize the bend range for the assigned channel.
     - 3.3 Add tests to verify these CC messages are emitted exactly once at the start of every track.

4. **Applying Pitch Bend per Note**
   - For every tone, we will calculate the pitch wheel value: `bend_value = int((offset_cents / 100) * 8192)`.
   - A `pitchwheel` message will be emitted immediately before the `note_on` message.
   - The `pitchwheel` message will consume the `pending_rest_ticks`, and the subsequent `note_on` will happen immediately (`time=0`).
   - We will explicitly emit a `pitchwheel` message for *every* note, including notes with a `0` `offset_cents`. This guarantees the bend is correctly reset for perfect semitones and doesn't incorrectly inherit the bend from the previous note.
   - **Implementation Steps:**
     - 4.1 Update `save_score_to_midi` to calculate `bend_value` from `pitch.offset_cents`.
     - 4.2 Inject the `pitchwheel` message into the `voice_track` immediately before `note_on`, passing `pending_rest_ticks` to the bend message and setting the `note_on` time to 0.
     - 4.3 Add tests asserting that a `pitchwheel` message precedes every `note_on` message and carries the correct wait time.
     - 4.4 Add tests asserting that the pitch bend value is correctly reset to 0 (center) when a standard diatonic note is played after a microtonal note.
     - 4.5 Add test for sequential rests: Verify `pending_rest_ticks` accumulates correctly across multiple consecutive rests before being consumed by the next `pitchwheel` event.
     - 4.6 Add test for leading silence: Verify RPN headers are emitted at `time=0` and the leading rest duration is correctly applied to the first `pitchwheel` message.
     - 4.7 Add test for bend clamping: Verify that calculated `bend_value` is clamped to the valid `-8192` to `8191` range to prevent invalid MIDI data.

## Technical Caveats
- **Pitch Range:** Standard MIDI note range is 0-127. Frequencies outside this range will be clamped.
- **Microtonal Resolution:** Pitch bend provides 14-bit resolution. Over a ±1 semitone range, this provides extremely high precision (much finer than 1 cent).
- **VST Compatibility:** Not all VST instruments respond to RPN messages for pitch bend range. Users may need to manually set the pitch bend range to ±1 semitone in their VST if it ignores the MIDI RPN messages.

## Future Experiments & Stretch Goals
- **Intentional Pitch Glitching:** As a future experiment, we may explore allowing voices > 15 to preserve their microtonal offsets while sharing channels. This would result in "pitch glitching" where voices fight over the channel's pitch wheel, potentially creating interesting emergent acoustic artifacts. This is currently excluded from the main implementation to ensure predictable playback.
